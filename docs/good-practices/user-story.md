# Buenas prácticas para crear User Stories

## 1️⃣ Enfocarse en el usuario

* Escribir **desde la perspectiva del usuario final**, no de la organización.
* Usar **personas** para entender objetivos y necesidades concretas.
* Ejemplo: docente, padre, administrador, niño (observador).

## 2️⃣ Mantener un formato simple y conciso

* Formato estándar:

```
Como [tipo de usuario], quiero [acción que desea realizar] para [beneficio que espera obtener].
```

* Evitar detalles técnicos o instrucciones de implementación.
* Mantener las historias **pequeñas y entregables en un sprint**.

## 3️⃣ Priorizar y estimar

* Priorizarlas según **valor para el usuario y para el negocio**.
* Estimar el esfuerzo necesario para implementarlas.
* Usar la priorización para decidir qué entra en cada sprint.

## 4️⃣ Fomentar la colaboración

* Escribir historias **colaborativamente con el equipo de desarrollo**.
* Usarlas como **punto de partida para conversaciones más profundas** sobre funcionalidad y diseño.

## 5️⃣ Incluir criterios de aceptación

* Definir condiciones claras para considerar la historia como “terminada”.
* Formato recomendado:

```
Dado que [contexto inicial], cuando [acción realizada], entonces [resultado esperado].
```

* Evitar centrar los criterios en **cómo construir** la funcionalidad.

## 6️⃣ Mantener las historias actualizadas

* Refinar y evolucionar las historias a medida que el proyecto avanza.
* Técnicas útiles: **User Story Mapping** para organizar y priorizar.

# Estructura básica de una User Story

1. **Título**

```
Como [rol del usuario],
quiero [acción que desea realizar],
para que [beneficio que espera obtener].
```

2. **Descripción**

* Texto conciso en lenguaje natural que explique la funcionalidad.

3. **Criterios de aceptación**

* Condiciones específicas que definen cuándo la historia está completa.
* Ejemplo:

```
Dado que el usuario está en la pantalla de asistencia,
cuando registra la asistencia de un niño,
entonces la asistencia se guarda en la base de datos y se notifica al padre.
```

4. **Notas adicionales**

* Información útil para desarrollo, diseño o contexto.

5. **Tareas**

* Lista de subtareas necesarias para completar la historia.

6. **Historias relacionadas**

* Referencias a otras historias que estén conectadas.
